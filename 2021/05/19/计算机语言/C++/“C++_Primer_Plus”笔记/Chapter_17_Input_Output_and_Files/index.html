Chapter_17_Input_Output_and_Files

对 C++ 输入和输出的概览

大多数计算机编程语言都会在语言的内部构建输入和输出方法，但是 C 和 C++ 都没有把输入和输出做到语言里面。

C语言本来想在 I/O 函数设计上，给开发者留有空间，以更好地使输入输出契合目标计算机的硬件环境。实际上，大多数的开发者都是基于原先在 Unix 环境下的库函数进行开发的。ANSI C 把这些 I/O 包正式化为“标准输入输出包”。C++ 也认可这个包，在头文件 cstdio 中可以使用包中的函数。

C++ 有自己的关于输入输出的解决方案，这些都在 iostream 和 fstream 中找到。

流和缓冲

C++ 程序把输入和输出都看作是字节流（stream of bytes）。

输入的时候，程序从输入流提取字节；输出的时候，程序往输出流插入字节。

每个字节可以代表一个字符，但更常用的是，字节们组合成字符或数值数据的二进制形式。也就是说，尽管输入输出是字节，但是不一定表示为字符。

流，可充当程序与另外一端的中介，这种方法使得 C++ 程序无差别对待键盘输入或文件输入。也就是说 C++ 根本不关心字节是哪里来的，类似地，C++ 也不关心字节往哪里去。

建立输入流，需要两个连接，流的两端各建立一个连接：文件端（file-end）和程序端（program-end）；对应地，建立输出流也是一样的。

此处的文件端是一个广义说法，其实文件端除了可以是文件之外，还可以是设备如键盘，甚至是另外一个程序；而程序端就是主要处理输入字节流的地方。

通常来说，处理输入和输出时可以利用缓冲（buffer），缓冲是一块内存，用作暂时存放传输于两端之间的数据。其最大作用是，匹配两端不同的信息传输速率。比如说，在输出的时候，因为一次硬盘读写太慢，所以需要缓冲暂时存放程序要送往硬盘的内容，在一定条件达成后再一次性写入硬盘。输入完成后，清理缓冲以便用于下一轮传输，这个过程叫作刷新缓冲（flush the buffer）。

C++ 程序通常会在用户按下 Enter 键的时候刷新输入缓冲，也就是说 Enter 被按下后，缓冲区中的字符才输送到 cin 这类输入流对象中，这也是为什么我们之前写的程序都需要按下 Enter 后再运行下一步；C++ 程序通常会在用户传送换行字符（是 endl，而不是 \n）的时候刷新输出缓冲。

而刷新缓冲的时机是因实现环境而异的，比如说，在即将在输入时：当程序执行到输入语句时，输出缓冲会被刷新。基于 ANSI C 的 C++ 实现应该都是这样工作的。

流，缓冲，还有 iostream 文件

在头文件 iostream 中，C++ 定义了一些类，其中有：

  • streambuf 类：与流和缓冲相关
    • 提供缓冲的内存
    • 填充缓冲
    • 访问缓冲内容
    • 刷新缓冲
    • 管理缓冲内存
  • ios_base 类：代表流的通用属性，比如说，流是否开启，该流是纯文本流还是二进制流等。
  • ios 类：基于 ios_base，包含了指向 streambuf 对象的指针成员。
  • ostream 类：派生于 ios 类，提供输出方法。
  • istream 类：派生于 ios 类，提供输入方法。
  • iostream 类：基于 istream 和 ostream 类，因此继承了输出和输入方法。

创建一个 ostream 类，就相当于：打开了输出流，创建了缓冲区，将其与流关联在一起。

istream 和 ostream 实际上是模板参数为 char 的模板专业化的 typedef。类似地，wistream 和 wostream 本质上也是模板参数为 wchar_t 的模板专业化。

C++ 的 iostream 类库会照顾到很多细节。比如说，包含头文件 iostream 就自动创建了 8 个流对象，4 个用于狭义的字符流，4 个用于宽字符流：

  • cin/wcin：对应标准输入流。默认下与标准输入设备相关联，该设备通常为键盘。
  • cout/wcout：对应标准输出流。默认下与标准输出设备相关联，该设备通常为屏幕。
  • cerr/wcerr：对应标准错误流。默认下与标准输出设备相关联，该设备通常为屏幕，而且该流没有缓冲，也就是说，一旦有错误，立刻传送到设备。
  • clog/wclog：对应标准错误流。默认下与标准输出设备相关联，该设备通常为屏幕，该流有缓冲。

为什么说，一个对象代表了一个流呢？因为这个对象负责许多与流相关的成员变量，拥有许多与流相关的操作。举例说，流的一端连接着程序，另一端连接着标准输出，cout 对象，和其成员 streambuf 对象，管理着经过该流的字节。

使用 cout 输出

当我们输送字节流到屏幕上的时候，我们希望每个字节都代表一个字符的值。也就是说，输出数字 -2.34 到屏幕上的时候，我们希望输出到屏幕的内容是字节‘-’、字节‘2’、字节‘.’、字节‘3’、字节‘4’、所对应的字符值，而不是单纯的 64 位浮点数的二进制数据。因此，ostream 类面临的一个重要任务就是，把数值类型的值，比如说 int 和 float，转化成字符流。

流状态

cin 或者 cout 对象包含了一个继承自 ios_base 类的数据成员，其可描述流状态（stream state）。

描述流状态的变量，其类型为 iostate，实际上是位掩码（bitmask）类型，含有三个 ios_base 元素：eofbit，badbit，还有 failbit。

  • 如果 cin 操作遇到了文件的末端，则设置（set）eofbit；
  • 如果 cin 操作无法读到预期字符，则设置 failbit：遇到 I/O 错误，比如说尝试读取无访问权限的文件，或者写入有写保护的硬盘时，都可以导致设置 failbit；
  • 有未能诊断的错误导致流被破坏时，设置 badbit。
  • 不同的实现环境制定了不同的设置 failbit 和 badbit 的情形。
  • 当所有的位都清除为 0 的时候，流处于正常状态。

下面是一些 ios_base 的方法，能够反映或修改流状态：

I/O 和异常

假设输入函数设置了 eofbit，这会导致抛出异常吗？默认情况下是不会的。但是如果使用了 exceptions() 方法的话，就不一定了。

exceptions() 方法返回一个位域，该域内有三个位，对应 eofbit，failbit，和 badbit。

更改流状态只能通过 clear() 或者 setstate()，当然他们本质上都是使用 clear()。在修改流状态之后，clear() 方法会把当前修改过后的状态与 exceptions() 方法返回的位域相比，如果它们在同一个位上都设置了 1，那么就会抛出 ios_base::failure exception。

因为 exceptions() 设置的位域的语义是：哪个位设置为 1，则说明希望 clear() 不要把这个位设置为 1。exceptions 所维护的位域并不是描述流状态的位域。

get(char *, int, char) 和 getline(char *, int, char) 两个函数，默认情况下读取一整行。

这些被称作非格式化输入函数（unformatted input functions），因为他们只是简单地读取字符，而跳过了空白字符，也不作任何数据转换。

其中 get(char & ch) 会将读取到的字符放入到 ch 中；而 get(void) 会把读取到的字符转变成整型类型，作为返回值。

意外字符串输入

使用 get(char *, int) 和 getline() 的时候，有一些形式的输入会影响到流状态。eofbit 和 badbit 的设置与其他输入函数的行为是一致的：遇到 end-of-file，就设置 eofbit；遇到任何破坏了流的事情，比如说硬件损坏，就设置 badbit。但是还有两种特殊情况需要考虑：无输入和输入溢出。

get() 和 getline() 在读不到字符的时候，都会把空字符（null character）放入目的地址中，然后调用 setstate()，设置流状态为 failbit。但是这两者对“读不到字符”的定义却不太一样：

  • 一种无输入的可能性是：立刻读到 EOF；

  • 而对于 cin.get() 来说，还有一种可能：在输入的时候，除了按回车，什么也不输入。

    • 这对于 cin.getline() 来说是有输入的，尽管 cin.getline() 不保存换行符，但是其会丢弃该换行符，实际上是先读取再丢弃的，所以 cin.getline() 在这种情况下算有输入，但是 cin.getline() 也是会把空字符放入目的地址中，只是不设置 failbit。

    • 所以不能够指望 cin.getline() 在读到空行时会返回 false：while(cin.getline(temp, 80)){...}，应该添加条件：

对于判断输入溢出，两者之间也有差异：

  • getline() 方法会一直读字符，直到遇见 EOF，直到下一个字符是分隔符，直到已经存储到了上限。
    • 如果遇见 EOF，则设置 eofbit；
    • 如果遇到分隔符，则读取且丢掉；
    • 如果存储到达上限，则设置 failbit，除非下个字符是换行符。这句话的意思是，如果刚好抵达上限的时候，下一个字符是换行符，则可以看作输入了恰好多的字符；否则，判断为该行还有更多字符没法输入，即设置 failbit。
  • get(char *, int) 方法首先测试字符数，其次是 EOF，最后再判断是否遇上分隔符。
    • 测试字符满时，get() 不会设置 failbit；如果要判断是否因为输入溢出导致的输入停止，可以使用 peek() 方法，查看下一个输入的字符。如果 peek() 返回的是分隔符，则说明 get() 已经读取了一整行；而如果返回的不是分隔符，则说明 get() 在分隔符前停止了。
    • peek() 方法对 getline() 来说是没用的，因为 getline() 会把分隔符丢掉，这样就无法进行下一步的判断。

总结：

  • getline(char *, int)：
    • 如果没有读取到字符（分隔符也算被读取），则设置 failbit
    • 如果读到了最大限度的字符，而且流中还有分隔符之外的字符，则设置 failbit
  • get(char *, int)：
    • 如果没有读取到字符（分隔符不处理，不算读取），则设置 failbit

其他 istream 方法

其他的 istream 方法还有 read()、peek()、gcount()、和 putback()。

  • read()：读取一定数量的字节内容到目的地址
    • 不附加空字符，也就是不将输入转化成字符串。
    • read() 不是为了键盘输入的，而是在大多数情况下与 ostream 的 write() 一起用作文件输入和输出。
    • 返回调用对象的引用，可级联使用：cin.read(gross, 144).read(score, 20);。
  • peek()：返回输入流中的下一个字符，但是不提取。
  • gcount()：返回上一个非格式化提取方法所读取的字节数。
    • 需要注意的是，对于 >> 来说是没有用的，因为使用了提取运算符的 cin >> 是格式化提取方法，是对字符作了转换处理的。
  • putback()：往输入流中插回一个字符。
    • 接收一个参数，存储要插回的字符。
    • 返回 istream &，可级联使用。

附加到文件

二进制文件

当我们存储数据到文件的时候，我们可以以文本形式存储，也可以以二进制格式存储。比如说，对于浮点数 2.324216e+07，我们将其以文本形式存储，程序会将这 13 个字符存储到文件中；若我们将其以二进制形式存储，程序会将该浮点数在计算机内部的二进制值存放在文件中。

相较于文本形式的易读，二进制形式的优点在于：

  • 存储数字时更精确，因为其存储的是值的精确内部表示，而不存在转换错误或者四舍五入导致的精度变化。
  • 存储速度更快，因为没有额外的转化时间。
  • 二进制的格式通常来说占用更少空间。

但其缺点在于，转移到另一个系统的时候很可能会出问题：

  • 有些数据类型的占位大小在不同系统下有所差异
  • 有些数据类型的具体存储格式在不同系统下有所差异

随机访问

随机访问意味着直接移动到文件的任何位置，而不是按顺序移动。（怎么着就不是顺序移动了呢？？？）

该访问方法通常用在数据库文件上。程序会维护一个单独的索引文件，该索引文件会记录数据文件中数据的位置。这样我们就可以直接跳到数据文件的具体位置，然后进行读取或修改。

当文件是由一堆大小相同的记录组成时，该方法最容易实现，每个记录都代表着一连串相关的数据，比如说，存储介质中的一个结构体可看作是一个记录，而结构体中的成员则为一连串相关的数据。

如果要对某文件同时读写，那么可以创建 fstream 对象。fstream 对象有两个缓冲，一个用于输入，另一个用于输出。指向输入缓冲的指针移动时，指向输出缓冲的指针也会移动，反之亦然。总之，fstream 对象的两个文件指针会始终指向同一位置。
