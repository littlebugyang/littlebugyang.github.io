string 类

string 类是在头文件 string 中声明的。需要注意的是，头文件 string.h 和 cstring 中的字符串函数适用于 C 风格的字符串，而非 string 类。

使用一个类的关键是悉知其公共接口。

构建一个字符串

string 类中的 string 实际上是模板专业化 basic_string< char > 的 typedef。

size_type 是基于实现环境的整型类型，在头文件 string 中定义。

string 类定义了 string::npos 作为 string 的最大长度，通常来说，该值为 unsigned int 的最大值。

以下用 NBTS（null-byte-terminated string） 作为“以 NULL 字节终结的字串”的缩写，这也是传统的 C 风格字符串。

下面看几个 string 的构造函数：

string(const char * s, size_type n);

用 NBTS 初始化该 string，用该 NBTS 的前 n 位；尽管 n 超过了 NBTS 的大小，其位上的数据照单复制，翻译成字符。

template < class Iter > string (Iter begin, Iter end);

以 [begin, end) 上的内容构造该 string。

string (const string & str, size_type pos, size_type n = npos);

从 str 的 pos 开始，以 [pos, n) 上的内容构造该 string 对象，默认 n 为最大长度。

以上 C 风格字符串和 string 类字符串的输入，主要区别在于：

  • 使用区别：后者可以自动调节目标字符串对象的大小，用以存储字符
  • 设计区别：前者使用到的方法是 istream 类中的，而后者使用到的方法是单独成体系的（string 头文件）。这也是在处理 C 风格字符串时要用 cin 作为调用对象，而处理 string 字符串时却把 cin 作为参数的原因。

虽然 string 对象可以根据输入调节字符串长度，但是这也是有限制的：

  • string 类的内置限制，也就是 string::npos，代表着该类所允许最大的字符串长度。
  • 程序可用内存大小。如果程序可利用的内存不足以放下更多字符，则即便字符串大小尚未达到 string::npos，也会输入失败。

string 类的 getline() 函数从输入中读取字符，并存储到 string 对象中，除非以下三种情况之一发生：

  • 读到文件末端（end-of-file）。此时，输入流对象的 eofbit 被标记，意味着 fail() 和 eof() 方法会返回真。
  • 读到分隔符。该分隔符会被从输入流中丢弃，不被存储。
  • 读到的字符数超过了最大可能值（string::npos 和 可用内存字节数中的较小者）。这时 failbit 会被标记，意味着 fail() 方法会返回真。

输入流对象有一个会计系统（accounting system，不过这翻译是什么鬼），用于追踪流的错误状态：

  • 设置 eofbit 意味着检测到文件末端
  • 设置 failbit 意味着检测到输入错误
  • 设置 badbit 意味着出现了未知错误，比如说硬件错误
  • 设置 goodbit 意味着一切尚好

用于 string 类的 operator>>() 函数也跟用于 C 风格的字符串的相似，只不过与之不同的是，string 中的 operator>>()，不是读到分隔符时就丢弃之，而是读到空白字符（white space character）就停下来，而且把该字符留在输入队列中。空白字符可以是 space，newline，或者是 tab，更普遍地说，应该是任何 isspace() 函数返回真的字符。

智能指针模板类

平时，在使用指针的时候，如果使用了 new 关键字，在堆中分配了内存，那么就要在适当的时机使用 delete，释放掉这部分内存。但是这种“just remember to”的编程风格经常会导致错误，而且，如果在 new 和 delete 中间有意外跳转，delete 还是不能够正常使用，这就会造成内存泄露。

指针变量一般为自动变量，在其所在的函数生命周期的末端时，变量会被销毁，但是其所指向的堆内存并不会。如果指针像类一样，有析构函数，该析构函数可以释放所指向的内存，这样就可以有效避免内存泄露了。基于这种想法，C++98 引入了 auto_ptr，而 C++11 引入了 unique_ptr 和 shared_ptr，并且弃用了 auto_ptr。

标准模板类

一个容器（container）就是一个单元，像是一个数组，可以持有多个值。STL 的容器是同质的，也就是说，STL 的模板中存放的都是同样类型的值。

迭代器（iterator）是让你可以遍历容器的对象，是指针的泛化。

函数对象是行为像函数的对象：可以是类对象或者函数指针（函数名）。

STL 不是 OOP 的例子，而是另一种编程范式的代表：泛型编程。

更多要对向量做的事情

vector 模板类并不对所有的常用操作都定义方法（search、sort、shuffle 等等），STL 站在一个更为广阔的视野，把这些操作定义成了非成员函数。举个例子，STL 定义了一个所有容器类都可用的非成员函数 find()，而不是为每个容器类都定义各自的 find() 成员函数。

但是 STL 有时候也会为容器类定义成员函数，尽管已经有非成员函数可以完成相同的任务了。这样做的原因一般是：可以有针对该类的更加高效的算法。比如说，vector 类中的 swap() 会比非成员的 swap() 更高效；但是非成员函数的 swap() 允许你在两个不同种类的容器之间交换内容。

输入迭代器

输入这个术语是从程序的角度看的，也就是说，从容器到程序的信息流向，就是输入。所以，输入迭代器（input iterator）可被用于从容器中读取值。

对输入迭代器取消引用，必须使得程序可读，但不必可修改该容器值。所以，需要输入迭代器的算法并不会改变容器中元素的值。

输入迭代器必须允许访问容器内的所有值，支持 ++ 运算符，包括前缀和后缀版。

然而，输入迭代器并不保证第二次遍历的效果与第一次的一样；而且输入迭代器自增后，并没有保证之前的值还可以被取消引用。任何使用输入迭代器的算法都应该是单遍历算法，也就是不依赖于上一次遍历中的迭代器的值，甚至也不依赖于同次遍历中的迭代器早些时候的值。

需要记住的内容是：输入迭代器是单向的迭代器，可以增加，但不可以退后，且只可以读。

输出迭代器

输出意味程序输出信息到容器中。输出迭代器和输入迭代器几乎相同，除了在取消引用的时候，输出迭代器只可以用于修改容器的值，但是不能够读取容器的值。

虽然可写不可读有点奇怪，但是看一下 cout 就好理解了：cout 只能修改输出流中的字符，但是无法读取输入流的字符。

需要记住的内容是：输出迭代器是单向的迭代器，可以增加，但不可以退后，且只可以写。

前向迭代器

前向迭代器仅仅使用 ++ 运算符在容器内移动，所以，前向迭代器一次只可以走过一个元素。而跟输入和输出迭代器不同的是，每次使用前向迭代器，其走过一串值的顺序都是一样的。

尽管前向迭代器自增了，我们还是可以对该迭代器之前的值进行取消引用，得到相同的值。

前向迭代器允许我们读和写数据，或者仅仅允许读（加上 const 限定符）。

双向迭代器

双向迭代器拥有前向迭代器的所有特性，而且支持 -- 运算符，包括前缀和后缀版本。

随机访问迭代器

随机访问迭代器拥有双向迭代器的所有特性，除此之外，还添加了其他运算符操作，以便于实现直接跳转到容器内任一元素的位置：

  • + 运算符，迭代器与整型数相加
  • - 运算符，迭代器减去整型数
  • +=，-=，迭代器加上或减去整型数
  • []，等价于迭代器加上整型数后取消引用
  • - 运算符，迭代器与迭代器相减，判断迭代器距离
  • <，>，<=，>=，迭代器之间判等或判先后

迭代器等级制度

以上五种迭代器，构成了等级制度。等级越高，如随机访问迭代器，具有等级更低的迭代器的全部功能。

一个算法如果是根据某特定迭代器写的，则程序可以使用该特定迭代器作为参数调用该算法，又或者说使用更高级的迭代器调用该算法。

所以，设置不同种类的迭代器的目的是：按最低要求写算法，可以使得最多的容器可使用该算法。

需要注意的是，以上不同种类的迭代器并不是定义好的类型，而是概念表征。事实上每个容器类都使用 typedef 定义了类作用域内的 iterator 类。

比如说，vector<int> 中有 vector<int>::iterator 类，但是文档会告诉你，该迭代器是随机访问迭代器。那么，该迭代器可用作所有算法的参数（如果参数类型是迭代器的话），因为随机访问迭代器是最高级的迭代器，无论算法需要什么样的迭代器，随机访问迭代器都拥有对应的功能。

又比如说，STL 实现了双向链表，其迭代器类型是双向迭代器，则该迭代器不可用于需要更高级别迭代器的算法。

概念，细化，和模型

STL 关于各种迭代器的特性，在 C++ 中是不可表达的。所以，上面说到的五种迭代器，并不是真实存在的某个类型，而是五套迭代器的实现准则，是概念（concept）。

如果你正在设计一个容器类，而且确实需要迭代器，则可以在 STL 中找到标准变体的迭代器模板（这句我也不懂）。

五种迭代器的概念可以有类似继承的关系，比如说，双向迭代器继承了前向迭代器的能力。但是我们不能够把 C++ 的继承机制应用到迭代器上。举个例子，我们可能把前向迭代器实现为一个类，而把双向迭代器实现为一个正常指针（but why not both classes），在 C++ 的语法条件下，双向迭代器是内置类型，是不可能继承于某类的。

有一些关于 STL 的文章把这样的概念上的继承称作细化（refinement），这样的话，双向迭代器可看作是前向迭代器的细化。

某一概念的具体实现被称作模型（model）。比如说，一个正常的指向 int 的指针就是概念随机访问迭代器的模型，同时，也是前向迭代器的模型，因为该指针实现了前向迭代器这个概念的所有准则。

但我们还需要考虑边界问题。rbegin() 和 end() 实际上返回的都是一样的值（“past-the-end”），rend() 和 begin() 也是（首元素）。按照一般思路：

  • 首先，rbegin() 和 end() 都不能解引用，所以，rbegin() 在作为范围开端时，如果对其解引用，会引发错误；
  • 其次，rend() 指向了首元素，而 rend() 在作为范围末端时，其指向的元素是不被访问的，这样就导致遍历提前结束。

反向迭代器通过“先自减，后解引用”，解决了以上两个问题。rbegin() 先自减，则指向“past”，而不是“past-the-end”，此时再解引用，就是向量末元素的值；rend() 先自减，则指向“prior-the-front”，而不是“front”，这样就保证遍历最后可以恰好到达“prior-the-front”的后面，也就是“front”。这样一来，遍历就完整了。

目前的 copy() 算法有一个问题，就是其复制过程不仅是覆盖操作，而且不负责检测可用容量，所以，如果强行复制过多的元素到向量中，程序很可能会由于内存问题退出。此时，使用另外三种插入迭代器，可以把复制过程转换为插入过程。插入添加新元素，而不覆盖已存在的数据，而且使用自动内存分配，保证新信息可以被存储。

容器的类型

STL 有容器概念（container concept）和容器类型（container type）。

容器概念是指笼统的、有名字的类别，比如说，容器、序列容器、关联容器。

容器类型是可以用来创建具体容器对象的模板。原先的 11 种容器类型是：deque，list，queue，priority_queue，stack，vector，map，multimap，set，multiset，还有 bitset。C++11 添加了 forward_list，unordered_map，unordered_multimap，unordered_set，还有 unordered_multiset；而且还把 bitset 从容器类型中移出，作为其他类型的模板了。

容器概念

没有什么容器类型是完全对应了基本容器概念的。但是，概念描述了所有容器类常用的元素。容器概念是一种概念上的抽象基类，之所以是概念上的，是因为容器类没有真的使用继承机制。也就是说，容器概念只是列出了一套条件，所有的 STL 容器类都应该满足。

容器是对象，可以存放其他对象，而这些被存放的对象都是同一个类型的。但我们也不是可以存放任意一种对象到容器中，可存放的对象的类型必须可被复制构造（copy constructable）和可被赋值（assignable）的。基本类型满足这些需求，而且类类型（class type）也满足这些需求，除非这些类定义了复制构造函数或赋值运算符为私有或受保护。C++11 细化了这些概念，添加了像可复制插入（CopyInsertable）、可移动插入（MoveInsertable）等术语。

序列容器

序列容器要求其元素以严格的线性顺序排列，也就是说，序列中要有第一个元素，也要有最后一个元素，而且除了头尾两个元素，中间的每个元素不仅有前驱，而且有后继。数组和链表就是序列的例子，而分支结构（如，树）不是。

序列容器中有一些类，比如 vector 和 deque，都实现了 a[n] 和 a.at(n) 的需求，这两者不同的是：后者会做边界检查，而且在试图访问边界外的元素时抛出 out_of_range 异常。

vector

vector 模板类是序列类型中最简单的，也是被认为最应该默认使用的容器，除非有其他容器能更加符合程序的要求。

deque

deque 模板类在头文件 deque 中声明，代表着双端队列，发音与 deck 相同。

跟 vector 一样的是，deque 支持随机访问；跟 vector 不一样的是，deque 在首端插入和删除只需要常数时间。

如果大多数的操作都是在序列的头和尾发生的话，那么我们应该考虑使用 deque。

尽管 deque 在首部操作的时间复杂度比 vector 要小，但 vector 在很多操作上要比 deque 快。

list

list 模板类代表着双向链表。

  • 跟 vector 一样，list 是可逆的容器。
  • 跟 vector 不一样，list 不支持数组标记和随机访问。
  • 跟 vector 的迭代器不一样，list 的迭代器在插入或删除操作后，还是指向原来的元素。

list 有许多方法。其中 insert() 与 splice() 的不同在于：insert() 插入的是原数据的拷贝，而 splice() 把原数据移动到指定地点。

splice() 方法还有移动单个元素和移动范围内元素的原型。

splice() 保持迭代器的有效性，也就是说，在 splice() 过后，尽管迭代器指向的是被移动的元素，在 splice() 完成后，迭代器依旧指向该被移动的数据，即便不是在原来的容器中。

unique() 方法仅仅把相邻的相同值缩减至一个单值。但是结合 sort() 的话可以实现全链范围内的值去重。只不过，list 的快速插入的代价是放弃随机访问，所以非成员函数的 sort() 是无法应用于 list 的，所以，list 模板类包含了一个成员函数，可执行针对于 list 的排序任务。

相关联容器

相关联容器（associative container）是另一个细化的容器概念。相关联容器把值（value）和键（key）关联起来，以便于根据键找到值。

对于容器 X，通常来说，X::value_type 指明在容器中存放的值的类型是什么；而对于相关联容器来说，X::key_type 指明用作键的类型是什么。

相关联容器的优势在于，它提供了对其元素的快速访问。同时，像序列那样，相关联容器允许你插入新元素，但是不能够指定插入的位置，其原因是相关联容器通常会使用一个特殊的算法，决定应该把元素放在什么地方，而且之后可以快速读取。

相关联容器通常实现成树的一些形式。树的节点形式使得添加和删除元素可以像链表那样简单，但是树的搜索效率要比链表的高。

STL 提供 4 个相关联容器：set，multiset，map，multimap。前两个可以在头文件中找到（之前分别可以在 set.h 和 multiset.h 中找到），后两个可以在头文件 map 中找到（之前分别可以在 map.h 和 multimap.h 中找到）。

其中最简单的是 set：其值类型与键类型是一致的，而且键是唯一的，更贴近实质地说，set 中的值就是键。

multiset 与 set 相似，不一样的地方在于，multiset 的键可以有两个或以上的值。

map 类型中，值类型与键类型是不一样的，而且键是唯一的，一个键对应一个值。

multimap 类型与 map 类型相似，不一样的地方在于，一个键可以关联多个值。

算法

STL 的算法很多都有一个共同的特征：使用迭代器标识数据范围，标识结果存放的位置。STL 算法函数的设计有两个主要的泛型组件：

  • 模板，用于提供泛型类型。
  • 迭代器，用于泛型表示获取容器中的数据。

算法组

STL 把算法库分成 4 组：

  • 非修改序列操作（nonmodifying sequence operations）
    • 不修改容器，比如，find()，还有 for_each()。
  • 突序列操作（mutating sequence operations）
    • 可修改容器，修改值、修改元素位置，比如说，transform()，random_shuffle()，copy()。
  • 排序和相关操作（sorting and related operations）
    • 排序算法，还有集合算法等。
  • 广义数值操作（generalized numeric operations）
    • 计算内积，计算和，计算相邻差异等。
    • 通常操作的容器为 vector。

前 3 组在头文件 algorithm 中（之前在 algo.h），而第 4 组因为是特定面向数值的，所以其在头文件 numeric 中（之前在 algo.h）。

还有一种给算法分类的方法是看算法结果的存储位置。有算法是就地完成工作的，另外的算法是在别处复制数据从而完成工作的。比如说，sort() 就是一个就地算法（in-place algorithm）；而 copy() 将范围内的元素复制到别处，为复制算法（copying algorithm）。

总结

算法使用模板，使算法本身泛型于存储对象类型；算法使用迭代器，使算法本身泛型于容器类型。

也就是说，使用模板，可以使得算法适用于不同的对象类型；而使用迭代器，可以使得算法适用于不同的容器类型。

STL使用术语概念（concept）来表示一套需求（requirement）。该概念的实际实现被称作为该概念的建模（model）。基于其他概念的概念称为细化（refinement）。容器类，就是容器概念的建模。
