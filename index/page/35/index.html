友元

类也可以为友元，也就是友元类。友元类的所有方法，都可以访问原类的私有和受保护成员。

类自己决定哪个函数、哪个成员函数、哪个类为友元，友元关系并不能从类外部决定，所以友元的存在并不违反 OOP 的 Hide the Data 的原则。

友元类

现欲构建电视类（Tv）和遥控器类（Remote），我们需要厘清这两者的关系。首先，它们不是 is-a 的关系：很明显电视不是遥控器，反过来也不可能；其次，has-a 的关系也不太成立：遥控器不拥有单独的电视，遥控器可以用于遥控多台电视机，反过来，电视机可以被多个遥控器遥控。

它们之间的交互更像是友元关系：遥控器需要访问电视的信息，或者修改电视的信息，而电视机不需要对遥控器进行操作，这是一个单向的数据流，非常适合使用友元。把遥控器类设置为电视类的友元类，即可完美建模该问题。

友元声明可以在类的任何范围：public、private、protected，效果都是一样的。

嵌套类

在另一个类中声明的类，被称作嵌套类。

对于包含类的外界来说，嵌套类只能在使用作用域解析符和嵌套类被声明在公共部分的情况下，被使用。

使用嵌套类的原因通常是：辅助包含类的实现，避免名字冲突。

嵌套类和访问

对嵌套类来说，有两种访问。首先，嵌套类声明的位置控制了嵌套类的作用域（scope）；其次，嵌套类也有 public、private、protected 等访问控制。

作用域

嵌套类在作用域方面，其实就像包含类的成员一样：

  • 如果在 private 部分，则包含类内可见，而外部不可见，派生类不可见；

  • 如果在 protected 部分，则包含类内可见，而外部不可见，派生类可见；

  • 如果在 public 部分，则包含类内可见，派生类可见，外部可见，不过外部需要额外的作用域解析符才能够访问：

异常机制

我们现在可使用异常机制来处理问题。处理异常有三个步骤：

  • 抛出异常
  • 使用处理器（handler）捕获异常
  • 使用 try 块

throw 关键字表示抛出异常。

程序使用异常处理器（ exception handler）来捕获异常。catch 关键字表示捕获异常，而异常处理器以 catch 关键字开头，紧随其后的是类型声明（type declaration），该声明表明捕获的异常类型。异常处理器也被称作 catch 块（catch block）。

try 块表明了异常可能会出现的地方，其后紧跟着一个或多个 catch 块。

执行 throw 语句有点像是执行 return 语句，但是与 return 语句不同的是，throw 语句执行后并不会只返回其调用者，而是一路返回，直到它回到一个 try 块。

catch 块看起来像是个函数定义，但当然不是。catch 后的 const char * s 表明该处理器匹配一个为字符串的异常。捕获到异常后，程序将执行 catch 块内的语句，接着退出该块。

若 try 中的语句没有抛出异常，则程序接下来会跳过所有紧跟其后的 catch 块。

如果抛出的异常没有 catch 块捕获呢？默认情况下，程序最终会调用 abort() 函数，但是这一行为是可以修改的。

函数可以抛出对象作为异常。这样做的一个很重要的优点是，我们可以使用不同的异常类型，用以分辨不同的抛出异常的函数和情况。

栈展开

现假设函数通过抛出的异常终结，而不是通过返回调用终结。程序开始释放掉栈中的内存，但是该动作并不是停止于栈中第一个返回地址（函数返回才在这里停止），而是一直到某个地址中含有 try 块，释放操作才停止。此时控制权交由异常处理器。这个过程，就叫作栈展开。

throw 机制的一个很重要的特性是：跟函数返回一样，栈上的所有自动类对象的析构函数都被调用了。但与函数返回不同的是，函数返回只处理那些被该函数放到栈中的对象，但是 throw 语句处理 try 块和 throw 语句之间的所有被函数们放到栈中的对象。如果没有这个特性，throw 就不会调用那些被中间函数放到栈中的对象的析构函数。

throw 机制中允许异常的再抛出：

在 catch 块中单独一个 throw 语句可以将当前处理的异常再抛出。如此，该异常将回到下一个可以处理该异常的 try-catch 组合。

更多异常特点

抛出异常语句处理栈中对象的方法，与函数返回语句不同，另外，抛出异常与函数返回不同的地方还在于：抛出异常时，编译器总是创建暂时的拷贝，尽管异常处理器已经明确了接收引用。

编译器这样做的原因有两个：

  1. 抛出异常后，该抛出异常的函数中的对象，包括该异常本身，就要被销毁了。如果抛出异常像普通函数调用那样返回函数引用，那么到时候异常处理器只能接收到一个引用了 nothing 的变量。
  2. 基类的引用可以同时指向派生类的对象。如果抛出的是一个派生类的异常对象，那么在形参设置为基类引用的情况下，异常处理器可以正常捕获该异常，并且利用多态；如果形参设置为基类对象，虽然异常处理器也可以正常捕获，但是派生类的基类部分之外的内容会被剥离。

异常，类，和继承

异常误入歧途时

异常被抛出后，可能会造成两种问题：意外异常（unexpected exception）和未捕获异常（uncaught exception）。

意外异常是由异常规格引起的。在 C++11 之前，C++98 引入 异常规格，规定了函数可能抛出的异常。若抛出的异常类型与规格中的不同，则会导致意外异常现象的发生。在默认情况下，程序会中止。

未捕获异常是由 try-catch 组合的缺失导致的。异常被抛出后，如果没有合适的 catch 块能够捕获这个异常，或者根本就没有 try 块，那么该异常就会被当作未捕获异常。在默认情况下，程序会中止。

我们可以修改程序的应对方法，以在遇上这两种异常时，选择停止运行之外的出路。

运行时类型识别

运行时类型识别（runtime type identification）的目的是，给程序提供一个标准的方法，在运行时得知对象的类型。

RTTI 是为了什么

RTTI 如何工作

C++ 有三种组件支持 RTTI：

  1. dynamic_cast 运算符，从一个基类指针生成派生类指针，如果转换失败，则返回 0、空指针。
  2. typeid 运算符，返回标识某个对象的类型的值。
  3. type_info 结构体，存放某个类型的信息。

RTTI 只可以应用于有虚拟函数的类。

dynamic_cast 运算符

dynamic_cast 运算符应该是最经常使用的 RTTI 组件。该运算符不告知某指针指向什么类型的对象，而是告知该指针变量是否可以安全地被赋予某类型对象的地址。

  • 指针所指向的类型与该对象类型一致

static_cast

该运算符用于可以隐式转换的两个类型。既然是基于隐式转换，则其当然是可以完成一些隐式转换做不到的事情：支持单向隐式转换的反向转换。

如果单向隐式转换是可行的，则 static_cast 不仅可以执行该单向隐式转换，且可以执行反向的隐式转换。

在继承类的多态中，static_cast 相当于 upcast+downcast。

reinterpret_cast

reinterpret_cast 运算符用于固有的有风险类型转换。

有时程序员需要做一些依赖于实现环境的事情，使用该运算符可以使这种行为的回溯得更简单。（是不是可以理解为：如果出了问题就找该运算符？）

通常来说，这样的类型转换是用于低级的、依赖于实现环境的编程，而且不可移植。
