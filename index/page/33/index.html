  • 派生类可以使用基类非私有的方法
  • 基类指针和引用可以指向派生类对象

  • 可以把派生类对象赋值给基类变量，隐式调用重载赋值运算符方法

继承：一种“是……”的关系

C++ 有三种继承：public，protected，和 private。

Public 继承是最常见的形式，而且建立了 is-a 关系。

意思是说，派生类对象，应该同时也是基类的对象。

Public 继承不能建立以下关系：

  • has-a
  • is-like-a
  • is-implemented-as-a
  • uses-a

多态的公共继承

公共继承下，派生类对象可以直接使用基类的公共和保护方法，但是如果我们想让这些方法表现得不太一样呢？也就是说，让某个方法可以根据调用它的对象不同而表现不同。这种复杂的行为被称作：多态（polymorphic）。

在 C++ 中，有两个关键机制可以实现多态公共继承：

  • 在派生类中重定义基类方法
  • 使用虚拟方法

开发 Brass 类和 BrassPlus 类

为实现多态，可定义虚拟方法（virtual method）。

使用关键字 virtual 的话，程序基于引用或者指针所指向的对象类型，调用不同方法。

不使用关键字 virtual 的话，程序基于引用或者指针的类型，调用方法。

如果要在派生类中重定义一个基类方法，通常的做法是，把基类方法声明为 virtual。这样一来，派生类中的方法也会自动转化成 virtual，不过在派生类中也显式写出来会更好。

而且，通常基类要声明一个 virtual 的析构函数。这是为了保证派生对象被销毁时，析构函数能够按正确的顺序被调用。

类实现

virtual 关键字只需要在类声明中书写，在函数定义中不需要。

对虚拟析构函数的需求

如果基类的析构函数不加上 virtual 关键字，则在销毁派生类对象时，就会调用基类的析构函数，而不会调用派生类的，这样一来，派生类对象无法被正常销毁。

加上 virtual 关键字之后，销毁派生类对象时，就会调用派生类的析构函数，而派生类的析构函数结束后，基类的析构函数又会被自动调用，这样就以正确顺序完成了析构函数的调用。

静态和动态绑定

在源代码中，把函数调用转译成一段代码的行为，称作绑定函数名（binding the function name）。

在 C 中，每个函数名都对应着特定的函数；在 C++ 中，由于函数重载，该任务会变得难一点。但这些始终都是在编译期间就能完成的任务，称作静态绑定（static binding），或者早绑定（early binding）。

然而，virtual 方法的出现，使得情况变得更加复杂，编译器必须确保：在程序运行的时候，方法名可以对应正确的虚拟方法。这种绑定就叫作动态绑定（dynamic binding），或者迟绑定（late binding）。

指针和引用类型兼容性

在 C++ 中，动态绑定与指针和引用调用的方法有关。

将派生类的指针或引用转换为基类的指针或引用，称之为向上转型（upcasting）；相反，则称之为向下转型（downcasting）。

向上转型在公共继承的时候，是允许的，不需要显式强制类型转换。向上转型具有传递性，比如说，从 BrassPlus 派生出 BrassPlusPlus，那么 Brass 的引用和指针，可以指向 Brass，BrassPlus，BrassPlusPlus 的对象。

向下转型必须要有显式强制类型转换，而且使用有风险。这种限制与 is-a 关系相符，大多数情况下，is-a 关系是不对称的。

虚拟成员函数和动态绑定

为什么有两种绑定，为什么默认绑定为静态

为什么有两种绑定，为什么不动态到底：为了效率和概念模型。

为了效率很容易理解。动态绑定需要在程序运行时占用更多的资源。

概念模型这点有点模糊。其意思是，在设计类的时候，总有一些方法是仅需要基类拥有即可的，对这些方法使用静态绑定，意味着派生类不需要改写，实现代码与概念模型的统一。

总的来说，就是在在派生类中重写的方法，就加上 virtual 关键字，否则，没有必要。当然，类的设计并不是线性的，很多方法并不是一眼就能看出来需不需要加 virtual 的。

虚拟函数是如何工作的

C++ 规定了虚拟函数的行为，但是把其实现工作留给了编译器开发者。

编译器处理虚拟函数中常用的方法是：在每个对象中添加一个隐藏的成员，该成员指向一个存放函数地址的数组。这个数组被称作虚拟函数表（virtual function table, vtbl），该表中存放着该类声明的所有虚拟函数。

如果派生类给某个虚拟函数作了新定义，虚拟函数表则将该虚拟函数的名字绑定到新定义的代码；

如果派生类没有给某个虚拟函数新定义，虚拟函数表则保持该函数名与原代码的绑定；

如果派生类新增了一个虚拟函数，虚拟函数表则分配一个位置用于存放该函数名及代码地址。

需要强调的是，无论给类新增一个还是一百个虚拟函数，类对象中新增的隐藏成员只有一个，那就是指向虚拟函数表的指针。

结合例子：

  2. 在表中找到 showAll() 的函数地址
  3. 根据函数地址跳转到需要执行的代码段

总而言之，按以上方法实现使用虚拟方法，有以下的内存和执行时间开销：

  • 声明了虚拟方法后，该类的每个对象中都有变量，用于存储虚拟函数表的位置
  • 对于每个类，编译器都创建一张虚拟函数表
  • 对于每个函数调用，都需要查表这一额外步骤

关于虚拟方法，需要知道的事情

一旦某类的某个函数被定义为虚拟函数，则该类的派生类，及派生类的派生类等，都将拥有这个虚拟函数。

构造函数

构造函数不能够为虚拟函数。

调用派生类构造函数后，派生类构造函数会调用基类构造函数，也就是说派生类和基类的构造函数各自不同，不存在继承关系。

因为派生类不继承基类的构造函数，所以也就没有将构造函数定义为虚拟函数的必要。

析构函数

如果某类是基类，则其析构函数应该是虚拟的。

如果不定义基类的析构函数为虚拟函数，在用基类指针指向派生类对象后，用 delete 销毁基类指针，则只能调用基类的析构函数，而不能调用派生类的析构函数。比如说：

在静态绑定的情况下，delete pe; 只能够调用 ~Employee()，这样一来，就只能销毁 Employee 的部分，而在 Singer 内部其他的析构工作就无法完成。而如果在基类中定义析构函数为虚拟函数的话，这里会实行动态绑定，调用 ~Singer()，在 ~Singer() 结束后，还会自动调用 ~Employee()，以正确顺序完成销毁。

所以，对于基类来说，尽管它不需要析构函数，它也要把自己的析构函数声明为虚拟函数。而对于派生类来说，声明析构函数为虚拟函数也不会产生错误，只不过动态绑定会损失一点性能而已。

友元

友元不能够声明为虚拟函数，因为友元不是类成员。

无再定义

如果派生类没有再定义函数，则该类会使用基类的函数定义；若派生类为长链继承中的一部分，则派生类会使用该函数最近一次定义的版本。

访问控制：受保护

在面对外部访问的时候，protected 跟 private 一样，其域内的成员无法被访问；在面对派生类访问的时候，protected 域内的成员可以被直接访问。

尽管可以在 protected 内声明数据成员，但是最好只通过基类方法来访问基类数据。

protected 访问权限对成员函数来说是大有用处的，它给予了派生类访问非公共内部函数的权限。

抽象基类

当我们为两个类建模关系的时候，很有可能这两个类的属性和行为互有交叉，尽管在概念上它们可以被称作 is-a 关系，但是落实到代码的时候，总是会觉得不对劲。这时，可以使用 C++ 的抽象基类（Abstract Base Class，ABC），把两个概念类的共同属性行为放到抽象类中，再从这个抽象类继承出两个具体类（concrete classes）。

只要声明中有纯虚函数（pure virtual function），该类就是抽象基类。

抽象基类没有对应的对象。也就是说，不能够使用抽象基类生成对象，抽象基类，只能作为基类存在。抽象基类至少有一个纯虚拟函数。

C++ 允许纯虚拟函数拥有定义，尽管纯虚拟函数的目的就是使该函数在基类中不需要定义。

应用抽象类概念

抽象基类哲学

有一种学术观点，称：具体类，应该是那些绝不会充当基类角色的类。

ABC 要求其纯虚拟函数在任何具体派生类中被覆写，强制要求派生类遵守 ABC 所设定的接口。

其他类方法注意事项

构造函数注意事项

构造函数与其他函数不同的是，构造函数创建对象。这也是为什么构造函数不能被继承的原因。

继承，意味着派生类对象可以使用基类的方法。但是如果连构造函数都要继承的话，那么该对象甚至还要靠基类的构造函数创建自身，而与之相矛盾的是，调用派生类对象构造函数之后，才能调用基类的构造函数，由此生成基类对象。

使用 const

如果一个函数定义了自己找形参为指向常量的引用或指针，则该函数不能将此函数传递给其他函数，除非这个函数保证不修改该参数值。

公共继承注意事项

不能继承的东西

构造函数不能被继承。但是 C++11 提供了允许构造函数继承的机制。当然，默认行为还是构造函数没有被继承。

构造函数不能被继承。

赋值运算符不能被继承。但是赋值运算符的函数签名在每个类中都不一样，因为该重载函数有一个形参为类类型。

赋值运算符注意事项

能否将一个基类对象赋值给派生类对象呢？答案是：也许能。

  • 如果派生类定义了单参数构造函数，参数为基类对象，则可以利用该层转化实现赋值；
  • 如果派生类定义了赋值运算符，专门处理基类对象赋值给派生类对象，则可以；
  • 如果以上两点都没有的话，那么要实现该目的，只能使用强制类型转换（向下转换）。
